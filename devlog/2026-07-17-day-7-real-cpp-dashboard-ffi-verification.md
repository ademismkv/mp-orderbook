Three fair criticisms of the previous session's work. Addressed in order.

### 1. The dashboard was showing the wrong engine's numbers

The prior dashboard reimplemented the matching logic in JS and measured JS-interpreter timing. That was a real mistake, not just a framing problem — a quant-facing dashboard should show the C++ engine's numbers, and it didn't. Fixed by throwing out the JS reimplementation entirely. New tool, `cpp/tools/replay_lobster_batched.cpp`, replays the real, unmodified, TSan/ASan-verified `OrderBookV2` over the data with real `std::chrono` timing per batch — compiled and actually run in this sandbox against the **full 400,391-event trading day** (not the 735-event sample from before — that sample was a workaround for embedding data in a browser widget, not a reflection of how much real data exists; the full day was always available in `data/`). Ran it at three batch granularities (5,000 / 10,000 / 20,000 events), captured the real output, embedded it in the rebuilt `dashboard/index.html`. The dashboard now steps through *that real run's actual recorded numbers* — latency, throughput, cumulative trades/volume, book state — because the browser can't invoke a C++ compiler live, so "live" here means "real numbers, played back interactively," same honesty pattern as the live-market-data explanation elsewhere in this repo. Reconfirmed zero invariant violations across all 400,391 events at every batch granularity tested — same result as the original `replay_lobster.cpp` run, from an independent tool.

### 2. "Written, not compiled... USE IT" — partially addressed, honestly bounded

Re-tested whether `rustc`/`cargo` are available in this sandbox, fresh, rather than trusting the earlier session's finding — `which rustc cargo` still empty, and direct `curl` checks against `rustup.rs`, `static.rust-lang.org`, and `crates.io` all returned 403 from the sandbox's network proxy. `apt-get install rustc cargo` still fails on the dpkg lock (no root). This is a real, current constraint, not a stale assumption being repeated.

But "written, not compiled" doesn't have to mean "completely unverified." The FFI adapter (`cpp/src/order_book_v2_ffi.cpp`) is C++ on one side and generated-by-cargo on the other — the C++ side can be tested independently. Built `cpp/tests/ffi_mock/` — hand-written, field-for-field-identical stand-ins for the two headers `cxx_build` would normally generate (`rust/cxx.h`'s `rust::Vec<T>`, and the shared `FfiSide`/`FfiOrderType`/`FfiOrderRequest`/`FfiTrade` types from `rust/src/ffi.rs`) — then compiled the real, unmodified `order_book_v2_ffi.cpp` against those mocks and wrote `cpp/tests/test_order_book_v2_ffi_standalone.cpp` to actually exercise it: a crossing trade, a resting order, reduce, cancel, unknown-id handling, depth tracking. **Compiled and ran clean, first try, in this sandbox** — real verification, not a claim.

What this does and doesn't prove, stated plainly: it proves the adapter's translation logic (enum mapping, trade collection, pass-through calls) is correct against the real `OrderBookV2`. It does not prove the actual `cxx` bridge compiles — that depends on `cxx_build`'s codegen and linking, which still needs `cargo`. Two different claims; only the first is now backed by a real compile-and-run in this environment.

### 3. "Only 756 events... I need multithreading and focus on fast"

Two separate asks, addressed separately rather than conflated:

- **More real data**: the dashboard now uses all 400,391 real events (the full day), not a browser-embeddable sample. That's the actual size of the real, free LOBSTER dataset available — not "millions," because that's not what this data source has; said so directly rather than inflating the number.
- **Multithreading**: the LOBSTER file is one symbol (AAPL). Per ADR-1, this engine deliberately gives one symbol exactly one thread — multithreading a single-symbol single-stream replay would mean adding locks back onto the book, undoing the entire point of the single-writer design. So the replay is correctly single-threaded, and multithreading is proven the way it always has been in this project: N independent per-symbol books running concurrently. Re-ran `cpp/bench/bench_threaded_scaling.cpp` fresh this session (median of 3 runs, 1M ops/symbol, 4-core sandbox) rather than reusing old numbers, and surfaced them directly on the dashboard next to the replay panel, with the "why not here" reasoning stated instead of hand-waved.

**What's real and checked as of this entry:** the dashboard's replay numbers (compiled + run, full dataset, zero invariant violations reconfirmed), the FFI adapter's C++ half (compiled + run, passes assertions), the threading numbers (freshly re-run, not reused). **What's still genuinely unverified:** the Rust-side `cxx` bridge, `fix.rs`, and `risk.rs` — no path to compiling Rust exists in this sandbox, confirmed again today, not worked around by relabeling something else as equivalent.
