# Day 10 — verifying the FlatHashMap replacement, and closing a real fuzz-coverage gap

**Context:** `cpp/include/order_book_v2.h` and `cpp/src/order_book_v2.cpp` already referenced a `FlatHashMap<OrderId, IndexEntry>` replacing `std::unordered_map`, a `level_array_growths()` counter, and an opt-in `OBV2_PROFILE_BREAKDOWN` timing instrumentation — but `cpp/include/flat_hash_map.h` and `cpp/bench/bench_breakdown.cpp` themselves turned out to already exist too, fully written, including an honest account (in `flat_hash_map.h`'s own comments) of two real mistakes found and fixed while building it: a splitmix64 hash finalizer that measurably hurt throughput (destroys the cache-locality benefit of real sequential order ids), and a tombstone-based deletion scheme that caused real 40-60ms compaction stalls, both replaced (identity hash; backward-shift deletion, no tombstones). None of this had been verified end-to-end yet, and the codebase didn't compile as of the start of this pass — this entry is that verification pass.

**Verified, in order:**
1. **Compiles clean.** `order_book_v2.cpp` + `test_order_book_v2.cpp` build with no errors/warnings under `-std=c++20 -O3 -Wall -Wextra`.
2. **Unit tests: 10/10 pass**, unchanged from before the FlatHashMap swap.
3. **Original differential fuzz (v1 vs v2): 24/24 match** — 8 seeds x {5,000, 50,000, 200,000} ops, plus a 1,000,000-op run at seed 42. Zero mismatches.
4. **Real gap found and closed: the original fuzz workload never exercises the rebase path.** `fuzz/workload.h`'s `generate_ops()` keeps price within a fixed ±50-tick band forever — with the default 20,000-tick window, `ensure_index_for_price()`'s grow/rebase branch (and therefore `FlatHashMap::for_each_mut()`, which only that path calls) is *never* entered, at any seed or op count. That means the original 24/24 match, real as it is, said nothing about whether backward-shift deletion and the rebase path behave correctly together — the highest-risk correctness intersection in this whole change. Added `generate_ops_price_walk()` (unbounded price random walk) plus `fuzz_v1_rebase.cpp`/`fuzz_v2_rebase.cpp` (v2 side built with a deliberately narrow 200-tick window) to force real, repeated rebases. **24/24 match on this workload too**, with real rebase counts from 1 to 66 per run — genuinely exercising the gap, not just filling in a number.
5. **TSan and ASan threaded stress test: clean.** `test_threaded.cpp` under both sanitizers, checksums matching the unthreaded reference in all three builds (plain, TSan, ASan). No new concurrency concerns expected here since each thread owns a private `OrderBookV2`/`FlatHashMap`, and none surfaced.
6. **Real NASDAQ replay: identical to every previously-documented number.** 400,391 real events, 13,298 trades, 528,509 total qty, 0 invariant violations, 10,902 cancel misses, 33 reduce misses — byte-identical to the numbers documented before this change, confirming `FlatHashMap` produces the same real-world behavior as `std::unordered_map` did. New data point that didn't exist before: `level_array_growths() = 1` on this real trading day, even with the generously-sized 150,000-tick window — the rebase path is rare in practice, as ADR-2 always said, but that's now a measured fact instead of a claim.
7. **Benchmark: real, repeatable improvement.** Same `-O3`, same sandbox, 3 runs: p50 dropped from 125ns to 84ns, p99 from ~417-458ns to 375ns, throughput up from ~4.9-5.1M to ~5.8-6.0M ops/sec.
8. **`bench_breakdown.cpp` compiles and runs.** Per-phase split on the standard 2,000,000-op workload: `match()` dominates at ~27% of add()'s wall time (called on every single order), `ensure_index`/`arena_alloc`/`fifo_link`/`index_insert` each land in the 6.5-8% range (only called on the ~62% of orders that rest rather than fully cross). Only ~55% of wall time is accounted for by instrumented phases — the rest is loop overhead, RNG generation, and the timer calls' own cost, exactly as the file's own honest caveat says.

**Kept:** the project's standing rule — real bugs found via actual compilation/execution get documented plainly, not smoothed over. The two mistakes in `flat_hash_map.h`'s history (bad hash choice, bad deletion scheme) are left in that file's comments rather than cleaned up, since "here's what didn't work and why" is more useful to a reader than a tidy success story.

**Why this matters beyond just "faster":** this closes out three of the ten items from the optimization punch list that started this thread of work — replace `unordered_map` (done, measured), reserve/fix `levels_` growth (measured via `level_array_growths()`, confirmed rare in practice, now with real fuzz coverage of the path), and per-order time breakdown (`bench_breakdown.cpp`, working). The hardware-counter and thread-pinning items (from the same list) were handed off to the user's own machine instead, for the reasons in devlog day 9b — this entry is the part of that list that didn't need real hardware to verify properly.
