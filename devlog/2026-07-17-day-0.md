## 2026-07-17 — Repo scaffold + architecture decisions

**Asked for:** repo skeleton (CMake + Cargo), a curated papers list, and concrete architecture decisions (not just options) for the Rust/C++ split, threading model, and order book data structure — via Claude in Cowork, working from a plan already sketched by a separate personal AI assistant earlier the same day.

**Got:** working CMake+GoogleTest-style skeleton wired to the existing v1 `OrderBook` (std::map/deque, price-time priority, 4 passing tests, originally written in an earlier session); a Rust sidecar skeleton with `cxx` noted but not yet wired in; four ADRs (single-writer-per-symbol threading, array+intrusive-list order book replacing std::map, cxx-for-control/ring-buffer-for-data FFI split, explicit v1 scope guard).

**Kept:** the v1 C++ matching algorithm's core logic (price-time priority, maker's-price fills, FIFO within a level) — it was sound. Kept the phased plan from the earlier planning session; this entry doesn't change *when* things ship, only sharpens *what* gets built.

**Changed:**
- flagged one real inconsistency in the plan — "lock-free order book" (Week 3 item) was the wrong framing; ADR-1 reframes it as lock-free *queues into and out of* an ordinary single-threaded per-symbol book, a smaller and more tractable problem than making the book itself lock-free.
- **found and fixed a real correctness bug while wiring up the actual build** (this only surfaces when you compile and run, not when reading the code): the original `match()` took `Order taker` **by value**, so `add()`'s own copy of `taker.qty` was never updated after a fill. Result — a taker order that filled *completely* would still get pushed onto the book again at its full original quantity, because `add()` checked its own stale `taker.qty > 0`, which was always true (never decremented). None of the original 4 tests caught this because they only assert on the returned trades list, not on book state after a full fill. Fixed by passing `taker` by reference into `match()`. Added a 5th test (`test_full_fill_does_not_rest`) that checks `depth()`/`best_bid()`/`best_ask()` after an exact full fill — this is the regression test that would have caught it.
- separately, the original code didn't compile as written: `bids_` and `asks_` are different `std::map` template instantiations (`bids_` uses `std::greater<double>` for descending order), so `auto& book = cond ? bids_ : asks_;` fails to typecheck (ternary requires both branches to produce the same type). Replaced the three ternary-based book selections with explicit branches / a small templated helper (`match_against<Map>`) instead of duplicating the match loop per side.

**Wrong / open:**
- the "4 tests passing" claim in the earlier planning note (`2026-07-02-matching-engine-tutorial.md`) was apparently never actually compiled — this is worth remembering generally: an AI-produced code sample that *reads* correct and has plausible tests is not the same as one that's been built and run. Actually compiling was what caught both bugs above.
- the plan said pre-trade risk is "called synchronously from C++" without stating whether that's actually necessary — ADR-3 flags this as unresolved and recommends the simpler async pre-check version for v1, but this is a real decision that needs a deliberate answer, not a default.
